## 一、产品定位与核心目标

本产品定位为**云端通用个人助手Agent**，解决的核心问题是：用户在日常生产生活中有大量重复性、多步骤的任务，现有工具要么只能"告诉你怎么做"，要么需要用户手动逐步操作，缺少一个能真正"动手做完"的智能助手。

产品的终极目标是**越用越懂你**——不是一个固定能力的工具，而是随着用户使用不断沉淀偏好、习惯、工作规范，逐渐演化成专属于这个用户的私人秘书。

在开始拆解各个模块之前，先理解整个产品的能力演进逻辑，这是所有后续决策的底层依据：

```
LLM知道什么   →  记忆 + RAG 扩展知识边界
LLM能做什么   →  Function Call + MCP + Skill 扩展行动能力
LLM怎么做得好  →  Agent Loop + Multi-Agent + Harness 保障质量
```

这三层不是一次性设计完的，而是随着实践不断暴露痛点，再催生出新方案。下面按这条逻辑线逐一拆解。

---

## 二、基础能力层：大模型本体

### 2.1 产品决策背景

2022年底ChatGPT出现，LLM本质是在海量文本上做next token prediction，训练完成后像一部"万能百科全书"。但很快暴露出两个核心缺陷，这两个缺陷直接决定了后续所有模块存在的必要性：

**缺陷一：无记忆**
每次API调用都是无状态的，用户上一句说"我叫张三"，下一次请求不带这条信息，模型完全不认识你。对个人助手产品来说，这意味着用户每次都要重复交代背景，体验极差。

**缺陷二：知识静态**
模型知识截止到训练日期，今天的天气、公司内部文档、最新业务数据一概不知。对于要处理真实业务的助手来说，这是根本性的障碍。

这两个问题分别催生了记忆模块和RAG，是整个产品架构的起点。

## 三、记忆模块

### 3.1 产品价值

记忆模块解决的是**用户重复表达成本**的问题。用户不应该每次开新对话都要重新交代"我是后端工程师、我喜欢英文注释、我们项目用FastAPI"。记忆模块让这些信息只说一次，永久生效。

衡量记忆模块好坏的核心指标只有一个：**能否减少用户的重复表达**，而不是"能存多少"。这是一个反直觉但经过验证的结论——几千字符的有界文件记忆，往往比复杂的向量检索系统更实用。

### 3.2 三层记忆架构

**第一层：短期记忆（会话内messages数组）**

当前对话的上下文，存在内存里，会话结束即消失。核心问题是上下文窗口有上限（从早期4K tokens到现在128K-1M），聊多了装不下。

五种递进策略应对窗口压力：

| 策略 | 原理 | 信息损失 | 适用场景 |
|------|------|----------|----------|
| 滑动窗口 | 只保留最近N条消息 | 高（早期信息全丢） | 简单场景、token预算紧张 |
| 摘要压缩 | LLM对历史消息生成摘要 | 中（丢失细节保留要点） | 长对话场景 |
| 分层保护 | 保留首尾消息，压缩中间 | 中低（关键信息受保护） | 任务型对话 |
| 结构化摘要 | 按模板字段压缩 | 低（保留结构化要素） | 复杂Agent场景 |
| 虚拟分页 | 仿OS内存管理，按需换入换出 | 极低（可随时检索回来） | 超长对话 |

本产品采用**结构化摘要**方案，在75%上下文窗口时触发，具体压缩流程在Agent Loop章节详细展开。

**第二层：长期记忆（跨会话持久化文件）**

分三个文件存储，分开的原因是更新频率和策略不同：

- `MEMORY.md`（2200字符上限）：存环境事实和项目约定，如"Python版本3.11"、"数据库配置在config/test.yaml"，相对稳定
- `USER.md`（1375字符上限）：存用户画像和偏好，如"偏好英文风格输出"、"后端工程师熟悉Java和Python"，随对话演进更新
- `db`（不限字数）：存会话历史，用于跨会话上下文恢复

字符上限的设计是刻意的——强制信息精炼，避免记忆文件无限膨胀占用上下文。

**第三层：预取注入（每轮对话前的动态注入）**

长期记忆不是一次性全量塞进system prompt，而是每轮对话开始前预取相关记忆，以`<memory-context>`围栏的形式动态注入messages数组。

为什么用围栏而不是直接写进system prompt？两个原因：
- system prompt是静态的，记忆每轮可能不同，放里面无法动态更新
- 围栏明确告诉LLM"这是记忆上下文，不是用户说的话"，避免语义混淆

注入格式：
```
<memory-context>
[System note: The following is recalled memory context, 
NOT new user input. Treat as authoritative reference data — 
this is the agent's persistent memory and should inform all responses.]

## User Profile
- 偏好英文风格输出
- 后端工程师，熟悉Python和Java

## Environment Facts  
- Python版本: 3.11
- 项目使用 FastAPI 框架
</memory-context>
```

关键工程细节：围栏消息在每轮循环开头要先移除上一轮的，避免累积。压缩前也要先移除围栏，避免把注入内容当成真实对话内容压进摘要造成污染。压缩完成后，下一轮循环会重新预取注入最新状态。

**流式输出过滤**：LLM在流式输出时可能把`<memory-context>`标签带出来，直接展示给用户就泄漏了内部信息。为此设计了`StreamingContextScrubber`，是一个有状态的流式文本清理器。难点在于标签可能被拆分在多个streaming delta里（如`<memory`在一个delta，`-context>内容</memory-context>`在下一个delta），所以需要缓冲区处理跨delta的标签检测，不能单个delta独立处理。

### 3.3 市场主流记忆方案对比

在设计本产品记忆系统前，有必要了解业界主流方案的取舍：

| 方案 | 存储方式 | 检索方式 | 自动化程度 | 适用场景 |
|------|----------|----------|------------|----------|
| Claude Code | Markdown文件系统（分类文件） | 全量注入system prompt | Agent主动写入 | 开发者工具 |
| ChatGPT Memory | 服务端数据库（事实陈述句列表） | 语义相关性选择注入 | 全自动提取 | C端消费者产品 |
| MemGPT/Letta | 分层虚拟内存（Core+Recall+Archival） | Agent自主搜索 | Agent自管理 | 超长对话/复杂任务 |
| Mem0 | 向量DB + 图DB | 向量+图检索 | 全自动提取+去重 | 通用AI应用 |
| Zep | 向量+实体图+时序 | 混合检索（语义+实体+时间） | 全自动（摘要+实体抽取） | 企业级应用 |
| LangGraph | 可配置多后端 | 可配置（全量/检索/摘要） | 需编排 | 框架开发 |

**本产品选型**：借鉴Claude Code的文件系统方案，结合异步后处理提取（类Mem0思路），在简洁性和智能性之间取得平衡。

### 3.4 写入机制：两种方案的取舍

**方案A：LLM主动调用memory工具（基础方案）**

memory是一个标准的function call工具，LLM在对话中识别到值得记忆的信息时主动调用。支持三种操作：

```
add    → 新增记忆
replace → 用old_text定位旧记忆后替换（处理信息更新和冲突）
remove  → 删除某条记忆
```

写入触发时机写在工具description里引导LLM判断：用户纠正了你、用户分享了偏好、发现了环境事实、学到了项目约定等情况下主动写入。

写入前必须做**安全扫描**，防止记忆投毒攻击（恶意内容通过工具调用污染记忆，影响后续所有对话行为）。

**方案B：异步后处理提取（更智能的补充方案）**

纯靠LLM主动调工具有缺陷：LLM专注执行任务容易漏掉顺带说的信息，不同模型对"值得记忆"的判断标准不一致。

更可靠的做法是每轮对话结束后，**异步起一个专门的LLM调用**做记忆提取，和主流程完全解耦，不阻塞用户响应。提取器有专门的判断prompt，明确定义什么值得记忆（用户偏好、身份背景、纠正行为、环境事实）、什么不值得记忆（单次任务内容、容易重新发现的信息、临时性指令）。

**产品建议**：两种方案结合使用最稳健——主流程保留memory工具（实时性），每轮结束后异步提取（兜底补漏）。

### 3.5 记忆系统的五大核心挑战

| 挑战 | 问题描述 | 解决方案 |
|------|----------|----------|
| 上下文占用 | 记忆注入占prompt空间 | 渐进式加载、选择性注入、摘要压缩 |
| 相关性检索 | 如何找到当前最相关的记忆 | Embedding语义检索 + Reranking + 元数据过滤 |
| 记忆过时 | 用户偏好会变 | 时间戳衰减、冲突检测更新、周期性验证 |
| 矛盾处理 | 新旧信息冲突 | Last-write-wins / LLM仲裁 / 保留两者标记冲突 |
| 安全威胁 | 记忆注入攻击 | 写入前安全扫描、敏感信息脱敏、审计日志 |

### 3.6 记忆的高级演进：三层学习机制

这是整个记忆模块最有产品价值的设计，也是"越用越懂你"的核心实现路径。

**核心思想**：被动记忆 → 主动技能

```
记忆：遇到相关场景才被检索到，可能漏检
规则：注入system prompt，每次都生效，但只是约束
技能：有触发条件+完整流程+工具支持，条件满足自动执行
```

类比：
```
记忆 = 新员工记笔记（翻到才想起来）
规则 = 公司制度上墙（每天都看到但只是约束）
技能 = 老员工的肌肉记忆（条件反射式执行，带完整SOP）
```

**L1：记住事实**（被动）

最基础的层次，遇到相关场景才被检索到，语义不匹配时可能漏检。

```
用户说："测试不要mock数据库"
→ 写入feedback记忆，下次遇到测试相关场景才会检索到
```

**L2：总结规则**（半主动）

当同一主题的feedback记忆积累到阈值，触发聚合，生成结构化规则注入system prompt，每次都生效，不依赖检索。

```
5条"代码规范"相关记忆积累后
→ 聚合成规则块注入system prompt
→ 禁止mock/函数不超20行/禁止any类型/copyright/import排序
```

**L3：形成技能**（全主动）

规则积累到一定程度，加上用户曾提过相关工具或流程，自动生成完整的SKILL.md文件。技能有触发条件+完整执行流程+工具支持，条件满足自动加载执行，不需要用户再提醒任何规则。

```
feedback记忆达到阈值 + 用户曾提过"事务回滚"、"config/test.yaml"
→ 生成 skill/integration-testing/SKILL.md
→ 包含：触发条件、禁止规则、完整工作流程
→ 用户说"帮我写个集成测试"，自动加载执行，不再需要提醒
```

三层的本质区别：

| 层次 | 能力 | 类比 | 实现方式 |
|------|------|------|----------|
| L1 记住事实 | 记住用户说了什么 | 新员工记笔记 | feedback/user类型记忆 |
| L2 总结规则 | 从多次事实中提炼规律 | 老员工总结经验 | 记忆聚合 → 行为约束注入system prompt |
| L3 形成技能 | 规则+流程+触发条件=自动化能力 | 专家制定SOP | 生成SKILL.md（含workflow+tools+references） |

**技能生成的两条实现路径：**

路径一：Agent主动提议（透明可控）
发现多次提醒类似的事，主动问用户"要不要整理成固定工作流"，用户确认后生成。优点透明可控，缺点需要用户参与。

路径二：系统自动转化（全自动）
后台定时任务扫描所有feedback记忆，用LLM做主题聚类，同主题达到阈值且置信度足够高时自动生成SKILL.md草稿。全自动，但需要置信度过滤防止生成大量低质量技能。

**防止技能爆炸**：阈值不能太低，否则产生大量细碎技能反而让系统变复杂。置信度过滤+用户审批是两道防线，缺一不可。

## 四、RAG模块

### 4.1 产品价值

记忆解决了对话连续性问题，但LLM能回答的仍然局限于训练数据。要让助手具备业务知识（公司内部文档、私有API、产品手册），就需要RAG：**先从外部知识库检索相关文档，把检索结果作为参考资料拼进prompt，让LLM基于这些资料生成回答**。

### 4.2 四个核心环节的产品决策

**环节一：文档分块（Chunking）**

分块质量直接决定检索效果。切太大检索精度下降，切太小上下文不完整。

| 策略 | 原理 | 优缺点 | 适用场景 |
|------|------|--------|----------|
| 固定大小 | 按字符/token数切割，设overlap | 简单快速，可能切断语义 | 快速原型、通用场景 |
| 递归分割 | 按分隔符层级递归（\n\n→\n→.→空格） | 比固定大小好，仍不语义感知 | 通用，LangChain默认方案 |
| 语义分块 | 计算相邻句子embedding相似度，骤降处切割 | 效果最好，计算开销大 | 语义完整性要求高 |
| 结构感知 | 按Markdown标题/HTML标签/PDF段落结构切割 | 保留层级关系 | 结构化文档 |
| Agentic | 用LLM判断每句话属于哪个主题 | 效果极佳，成本极高 | 最高质量要求 |

生产经验：chunk size典型值256-1024 tokens，overlap 10-20%。推荐递归分割+元数据增强（保留标题层级、页码、文件路径）。

**环节二：向量化（Embedding）**

将文本转为高维向量，语义相近的文本在向量空间中距离相近。

| 模型 | 维度 | 多语言 | 推荐场景 |
|------|------|--------|----------|
| OpenAI text-embedding-3-small | 1536 | 良好 | 英文为主、快速接入 |
| BGE-M3 (BAAI) | 1024 | 100+语言 | 中文场景首选，支持混合检索 |
| GTE-Qwen2 (阿里) | 多种 | 中英 | 阿里云生态、DashScope集成 |
| Jina-embeddings-v3 | 1024 | 多语言 | 长文档场景（支持8K上下文） |
| Cohere embed-v3 | 1024 | 100+语言 | 需要区分查询和文档embedding |

选型建议：中文场景首选BGE-M3或GTE-Qwen2（开源免费、中文效果极佳）；英文场景OpenAI text-embedding-3-small性价比最优。

**环节三：向量存储**

| 数据库 | 类型 | 数据规模 | 适用场景 |
|--------|------|----------|----------|
| ChromaDB | 嵌入式 | <百万 | 原型开发、小规模 |
| pgvector | PG扩展 | <千万 | 已有PG、不想引入新组件 |
| Qdrant | 独立服务 | 亿级 | 高性能要求 |
| Milvus | 分布式 | 百亿级 | 企业级大规模场景 |
| Pinecone | 全托管SaaS | 十亿级 | 不想运维基础设施 |

**环节四：检索优化**

单纯向量相似度检索在生产中远远不够，需要三种策略组合：

**混合检索**：向量语义得分 + BM25关键词得分加权融合。纯向量检索对专有名词（如AgentLoop、McpService）效果差，BM25精确匹配可以弥补。

**重排序（Re-ranking）**：先检索Top-50，再用Cross-encoder精排取Top-5。生产中re-ranking几乎是标配，精度提升10-30%。

**查询变换**：

| 策略 | 原理 | 解决问题 |
|------|------|----------|
| Query Rewriting | LLM改写查询 | 口语化查询检索效果差 |
| Query Decomposition | 复杂问题拆成多个子查询 | 多跳推理问题 |
| HyDE | 先生成"假设答案"，用假设答案的embedding检索 | 缩小query和document的语义鸿沟 |
| Multi-query | 同一问题生成多种表述分别检索后合并 | 提高召回率 |

### 4.3 RAG演进路线

```
Naive RAG（基础检索生成）→ Advanced RAG（优化检索链路）→ Agentic RAG（Agent驱动检索）
简单Top-K单次检索          混合检索+重排序+查询变换         Agent循环检索，自评估，多轮自适应
全信任结果                  结果过滤/验证                    自评估+纠错
```

**Self-RAG**：模型在生成过程中自评估——是否需要检索？检索结果是否相关？生成是否忠实于检索内容？不相关时丢弃结果重新检索。

**CRAG（纠正型RAG）**：对检索结果做质量评估，Correct就使用，Incorrect就丢弃并改用Web Search补充，Ambiguous两者都用。显著减少因检索质量差导致的幻觉。

**本产品采用Agentic RAG**，将RAG嵌入Agent循环，Agent自主决定何时检索、检索什么、结果是否足够，支持多轮检索、跨文档推理。

### 4.4 RAG效果评估：RAGAS框架

生产中需要量化RAG效果，RAGAS是目前最流行的评估框架：

| 指标 | 评估对象 | 含义 | 计算思路 |
|------|----------|------|----------|
| Faithfulness | 生成质量 | 回答忠实于检索内容的程度 | 答案中每个claim是否有context支持 |
| Answer Relevancy | 生成质量 | 是否真正在回答用户的问题 | 答案与原始问题的相关性 |
| Context Precision | 检索质量 | 检索结果中有多少是有用的 | Top-K中相关文档的排名位置 |
| Context Recall | 检索质量 | 是否检索到了所有必要信息 | ground truth中的信息被检索到的比例 |

## 五、Function Call & MCP

### 5.1 产品价值

RAG让LLM有了更多知识，但本质上仍是"信息提供者"——能告诉你答案，但不能帮你执行操作。Function Call解决的是让LLM从"说"到"做"的问题。

**关键认知**：LLM本身并不执行工具，它只负责"决策"（选择工具和生成参数），实际执行由应用侧完成。这种决策-执行分离的设计确保了安全性和可控性。

### 5.2 Function Call发展时间线

| 时间 | 里程碑 | 关键能力 |
|------|--------|----------|
| 2023.06 | OpenAI引入Function Calling | 单函数调用，模型输出函数名+参数JSON |
| 2023.11 | Parallel Function Calling | 一次响应可并行调用多个函数 |
| 2024.01 | 各厂商跟进 | Anthropic Tool Use、Google Function Calling、通义千问 |
| 2024.08 | Structured Outputs | 保证输出100%符合JSON Schema |
| 2024.11 | MCP发布（Anthropic） | 标准化工具协议，脱离具体LLM平台 |
| 2025.03 | MCP Streamable HTTP | 新传输层，支持无状态服务器，替代SSE |

### 5.3 Function Call工作流程

```
1. 应用定义工具Schema（名称、描述、参数类型）注入tools列表
2. 用户提问 → LLM判断需要调用工具 → 输出结构化调用指令（工具名+参数JSON）
3. 应用侧接收指令 → 实际执行工具 → 拿到结果
4. 结果回传给LLM → LLM生成最终自然语言回答
```

### 5.4 MCP：工具的标准化协议

Function Call的问题是工具定义硬编码在应用中，各厂商格式不同，换个LLM平台或新增工具都要改代码，工具无法复用。

MCP（Model Context Protocol，Anthropic 2024.11发布）建立了工具的标准化通信协议，底层用JSON-RPC 2.0，让工具即插即用。

三层架构：
```
Host（运行环境，如Claude Desktop）
  └─ Client（维护与单个Server的1:1连接）
       └─ Server（工具提供方，暴露capabilities）
```

MCP提供四类能力：

| 能力 | 说明 | 对应操作 |
|------|------|----------|
| Tools | 可执行的函数/操作 | tools/list, tools/call |
| Resources | 可读取的数据源（文件、数据库） | resources/list, resources/read |
| Prompts | 预定义的prompt模板 | prompts/list, prompts/get |
| Sampling | Server反向调用LLM | sampling/createMessage |

### 5.5 Function Call vs MCP 选型决策

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| 标准化 | 各厂商格式不同 | 统一协议标准 |
| 工具发现 | 硬编码在应用中 | 动态tools/list（运行时发现） |
| 可复用性 | 换平台需改代码 | 一个Server服务所有Agent |
| 额外能力 | 仅函数调用 | Resources + Prompts + Sampling |
| 部署复杂度 | 无额外组件 | 需要部署/维护MCP Server |
| 适用场景 | 工具少且固定（<5个） | 工具多、需复用、多平台支持（>10个） |

### 5.6 工具治理的四大生产问题

| 问题 | 表现 | 解决方案 |
|------|------|----------|
| 工具名拼写错误 | 大小写/格式错误/多余后缀 | 模糊匹配自动修复（Levenshtein距离/difflib，相似度≥70%） |
| 参数JSON格式错误 | 缺失括号、surrogate字符、尾随逗号 | 多策略修复（补括号→删尾随逗号→surrogate清理） |
| 嵌套参数展平 | LLM把{request:{query:...}}展平为{query:...} | 自动检测展平参数并包装回嵌套结构 |
| 上下文占用过多 | 50个工具schema占满prompt | Skill渐进式加载，按需注入 |

## 六、Agent Loop（核心执行引擎）

### 6.1 产品价值

有了工具之后，谁来编排整个流程？Agent Loop是这个"编排大脑"——一个不断让LLM思考下一步做什么、执行操作、观察结果、再决定下一步的循环，直到任务完成。

### 6.2 执行范式选型

| 范式 | 实现复杂度 | 适用场景 | 优势 | 劣势 |
|------|------------|----------|------|------|
| ReAct | 低 | 通用，简单到中等任务 | 灵活、实现简单、可观测性好 | 无全局规划，复杂任务易迷失 |
| Plan-Execute | 中 | 步骤可预见的确定性任务 | 全局规划合理，可展示计划 | 不灵活，中间出错难调整 |
| Plan-React | 中高 | 复杂但可拆解的任务 | 兼顾规划和灵活性 | 实现复杂 |
| LATS | 高 | 数学/代码/需探索的任务 | 可回溯，找到更优解 | Token消耗极高，延迟高 |
| Reflexion | 中 | 有明确成功标准的任务 | 从失败中学习 | 需多次尝试，成本高 |

**本产品选型**：ReAct主循环 + 可选Plan能力混合模式。Plan不是模式切换，而是ReAct循环中的一个普通工具（todo工具），简单任务零开销直接执行，复杂任务LLM自主决定是否调todo创建计划。

### 6.3 核心Loop设计：六重保障

整个Agent Loop的完整生命周期，按执行顺序：

```
用户请求进入
    ↓
① 迭代预算检查 → 当前迭代 < 最大迭代(默认90轮)？超出则生成工作摘要优雅退出
    ↓
② 中断检查 → 用户请求取消？保存状态立即退出
    ↓
③ 上下文压缩检查 → token ≥ 75%窗口？执行三步压缩 + 注入Todo状态
    ↓
   记忆预取 → 提取用户最近query → 预取记忆 → 注入<memory-context>围栏
    ↓
④ THINK：调用LLM（错误分类+重试保障）
    ↓
⑤ 空响应防护 → content和tool_calls都为空？计数器+1，超过2次终止
    ↓
   分支判断
   ├─ 有tool_calls → ACT：工具分发执行（名称/参数修复保障）→ 回到循环顶部
   └─ 无tool_calls → DONE：输出最终回答，同步记忆，保存会话，返回用户
```

保障及触发条件汇总：

| 保障 | 模块 | 触发条件 | 行为 |
|------|------|----------|------|
| ① 迭代预算 | IterationBudget | 达到上限（默认90轮） | 生成工作摘要→优雅退出 |
| ② 可中断 | _interrupt_requested | 外部线程设置标志 | 保存记忆→立即退出 |
| ③ 上下文压缩 | ContextCompressor | token ≥ 75%窗口 | 三步压缩+三级降级 |
| ④ 错误重试 | ErrorClassifier | API异常 | 分类→重试/压缩/终止 |
| ⑤ 空响应防护 | 连续计数器 | 连续2次空响应 | 注入提示引导→超限终止 |
| ⑥ 工具修复 | ToolDispatcher | 工具名/参数错误 | 模糊匹配修复→重新调用 |

### 6.4 上下文压缩：最复杂的子系统

**触发时机**：prompt token达到context_length × 75%时触发。为什么75%而不是90%？要给LLM的回复留空间（completion tokens也占窗口），加上压缩本身也要调LLM（需要token），预留25%余量。

**Token计算方式**：本地估算（字符数/4粗估，或接入tiktoken精确计算），不依赖API返回值。API返回的是上一轮消耗，下一轮还要拼入新消息，实际token数已变，本地估算反而更合适。

**压缩三步策略**：

Step 1：划分保护区
```
[HEAD保护区：前3条] [MIDDLE压缩区：所有中间消息] [TAIL保护区：后6条]
```
保护HEAD是因为system prompt包含Agent角色定义和工具列表，丢失则Agent"失忆"。保护TAIL是因为最近对话是LLM决策的直接依据，丢失会导致重复劳动。

Step 2：修剪MIDDLE区的工具输出
不同工具输出有不同的摘要策略，最大化信息密度：
```
terminal   → [terminal] ran `pytest tests/` -> exit 1, 42 lines output
read_file  → [read_file] read src/main.py from line 1 (5,234 chars)
write_file → [write_file] wrote to config.py (23 lines)
search_files → [search_files] content search for 'ERROR' in logs -> 3 matches
其他工具   → 短输出直接保留；长输出截取前150字+原始长度提示
```
这步是"物理压缩"，通常能减少40-60% token。

Step 3：LLM驱动的结构化摘要
将修剪后的MIDDLE发给LLM，按13个字段生成结构化摘要：
```
Active Task / Goal / Completed Actions / Active State / 
In Progress / Blocked / Key Decisions / Remaining Work / Critical Context...
```
摘要目标长度：min(max(最小200tokens, 原始内容×压缩比), 天花板2000tokens)

压缩完成后，最终messages结构：`[HEAD保护区] + [摘要消息] + [TAIL保护区]`，效果：100+条消息→约10条，token减少60-80%。

**安全细节**：摘要生成时明确要求LLM用用户的语言输出，且绝对不能在摘要中保留API Key、Token、密码（替换为[REDACTED]）。

**压缩失败时的三级降级**：

| 级别 | 操作 | 信息损失 |
|------|------|----------|
| 降级① | 尾部保护从6条降到3条，让更多消息进入压缩区 | 低 |
| 降级② | 直接删除最早10条工具结果 | 中 |
| 降级③ | 只保留system prompt + 最近3轮，相当于"重启" | 高 |

**压缩防抖**：如果最近几次压缩的平均节省率低于10%，说明已经压无可压，跳过压缩避免浪费LLM调用。

**Todo状态注入**：压缩完成后，把未完成的todo任务状态作为新消息注入回去，只注入pending和in_progress的任务（已完成和已取消的不注入，防止LLM误以为需要重做）。

### 6.5 错误分类与重试

不能一律"重试3次"，必须按错误类型分策略处理：

| 错误类型 | 典型表现 | 恢复动作 |
|----------|----------|----------|
| 瞬态错误 | 网络超时/500/连接断开 | 指数退避+随机抖动重试，最多3次 |
| 速率限制 | 429 Too Many Requests | 尊重Retry-After头或递增等待 |
| 上下文溢出 | 400 "context length exceeded" | 触发压缩后重试，不是普通重试 |
| 永久错误 | 401无效Key/模型不存在 | 终止+报告用户，重试无意义 |
| 服务过载 | 503/529 Overloaded | 重试+可切换提供商 |

**为什么要随机抖动**：多个客户端在失败后等待相同时间重试，会在同一瞬间再次涌入服务器（惊群效应/Thundering Herd）。加随机抖动使重试时间离散分布，避免雪崩。

实践案例：Anthropic官方SDK内置2次自动重试+指数退避；OpenAI SDK默认重试429/500/503错误。这些都是Harness的最小实现。

## 七、Skill系统（渐进式能力加载）

### 7.1 产品价值

工具数量增长带来两个核心矛盾：上下文占用（50个工具schema可能占50000 tokens），以及选择困难（工具超过20个后LLM选择准确率明显下降）。

Skill系统的核心思想来自操作系统的按需加载：不把所有代码加载进内存，需要时才加载。

Token效率对比：

| 方案 | 50个工具的Token消耗 | 效率 |
|------|---------------------|------|
| 传统全量注入 | ~50,000 tokens | 1x |
| Skill广告阶段 | ~5,000 tokens（每个skill约100tokens摘要） | 10x |
| Skill Load后 | +1,000~3,000 tokens（仅加载用到的1-2个） | 按需 |

关键洞察：大多数对话只会用到1-2个skill，没必要让LLM看到所有工具的完整定义。

### 7.2 SKILL.md标准格式

每个skill是一个独立目录，核心是SKILL.md文件，用YAML frontmatter声明元数据，Markdown body定义操作手册：

```
skill/log-diagnosis/
├── SKILL.md           → 必需：YAML frontmatter + Markdown操作手册
├── scripts/           → 可选：可执行Python脚本
│   └── parse_trace.py
├── references/        → 可选：参考文档（按需加载）
│   └── error-patterns.md
└── assets/            → 可选：模板、配置等静态资源
```

SKILL.md结构示例：

```yaml
---
name: log-diagnosis
description: 闲鱼圈子日志诊断助手。当用户需要排查圈子服务线上错误时使用。
tools:
  - query_sls_logs       # 声明需要的MCP工具
metadata:
  max_rounds: 10         # skill内ReAct最大轮次
mcp_servers:             # skill专属MCP服务器
  - name: sls
    url: https://mcp-sls.example.com/sse
---

# 操作手册
## 工作流程
1. 收集查询信息（时间范围、关键词、traceId）
2. 调用 query_sls_logs 查询日志
3. 加载 references/error-patterns.md 匹配历史错误模式
4. 输出结构化诊断报告

## 边界处理
- 无traceId时：引导用户提供请求时间范围
- 日志量过大时：缩小时间窗口后重试
```

Skill = SOP + 工具 + 资源。LLM不是在"猜"怎么用工具，而是在"按手册操作"。

### 7.3 四阶段渐进式加载

**阶段1：Advertise（启动时，一次性）**
扫描所有skill目录，解析SKILL.md的frontmatter，注入system prompt：
```
## Available Skills
- log-diagnosis: 闲鱼圈子日志诊断助手。当用户需要排查...
- code-review: 代码审查助手，确保代码符合团队规范...
```
开销：~100 tokens/skill。

**阶段2：Load（按需）**
LLM判断用户请求匹配某个skill → 调用`load_skill`工具 → 返回完整SKILL.md body → 同时触发MCP工具注入。开销：500-2000 tokens，但只在需要时发生。

**阶段3：Read（按需）**
skill操作手册中引用了参考资料 → 调用`read_skill_resource`读取。含路径穿越安全检查：用os.path.realpath比较，确保访问路径严格在skill目录内，防止LLM被注入后通过`../../../etc/passwd`读取系统文件。

**阶段4：Run（按需）**
需要执行脚本 → 调用`run_skill_script` → subprocess + 30秒超时保护 + cwd=skill目录。同样做路径穿越检查。

### 7.4 工具注册的条件性设计

Skill系统的工具注册不是全量注册，而是按能力有无条件注册：

```
load_skill          → 始终注册（只要有skill就需要）
read_skill_resource → 仅当存在含资源文件的skill时注册
run_skill_script    → 仅当存在含脚本的skill时注册
```

原因：每个注册工具都出现在LLM的tools列表中占用tokens。没有脚本的系统里注册`run_skill_script`是纯浪费（约200 tokens）。

### 7.5 MCP工具按需注入

用户加载某个skill时，系统从该skill声明的tools字段读取MCP工具名，动态注入到当前LLM调用的tools列表中，去重后append，不重复注入。

这样实现了"加入某个业务场景才加载该场景的MCP工具，避免上下文过长或LLM失焦"的产品目标。

### 7.6 Skill工具的四个来源

当skill执行期间LLM发起工具调用时，按优先级路由：

```
优先级1：scoped原生工具 - run_script（当skill目录下有scripts/子目录时提供）
优先级2：scoped原生工具 - read_resource（当skill目录下有references/时提供）
优先级3：skill专属MCP工具（SKILL.md的mcp_servers字段）
优先级4：全局MCP工具（SKILL.md的tools字段引用，兜底）
```

scoped工具优先级最高的原因：是skill作者特意为这个场景定制的，执行最快且最可靠。

## 八、Plan能力（嵌入ReAct的规划机制）

### 8.1 设计决策

传统方案需要在ReAct和Plan之间切换模式（两套循环逻辑），本产品的核心设计是：**Plan即工具**，todo工具和query_weather没有本质区别，都在同一个ReAct循环里调用。

```
简单任务: Think → Act(query_weather) → Answer
复杂任务: Think → Act(todo: 创建计划) → Act(step1工具) → Act(todo: 更新状态) → Answer
计划调整: Think(发现新情况) → Act(todo: 替换整个计划) → 继续执行新计划
```

优势对比：

| 维度 | 传统方案 | 本产品方案 |
|------|----------|------------|
| 简单任务 | 也要走Plan阶段（浪费） | 直接ReAct执行，零开销 |
| 循环逻辑 | 需要两套（ReAct loop + Plan loop） | 只有一套ReAct loop |
| 计划调整 | 需要专门的re-plan触发机制 | todo(merge=false)随时替换 |
| 压缩存活 | 计划可能被压缩丢失 | format_for_injection()自动注入 |

### 8.2 Todo工具设计

两种写入模式：

```
merge=false（默认）→ 用新列表完全替换，场景：创建新计划、推翻重来
merge=true        → 按id增量更新，场景：完成一步后更新状态、追加新步骤
```

每次返回完整列表而非diff，原因是LLM没有可靠的"记住上次工具调用结果"能力，全量返回保证LLM做下一步决策时有全局视角。

TodoStore是每个agent实例私有的，主agent和子agent各有独立实例，实现计划隔离，防止子agent完成的任务影响主agent的计划。

上下文压缩后，未完成的todo状态通过`format_for_injection()`自动注入回messages，只注入pending和in_progress的任务。会话被恢复时，从对话历史最近的工具返回值中重建TodoStore状态，确保计划不因上下文管理而丢失。

## 九、Multi-Agent（专家协作）

### 9.1 单Agent的三大困境

```
上下文膨胀 → 对话越长越容易"忘事"，即使压缩信息损失也不可逆
注意力分散 → 复杂任务容易"跑偏"，一个system prompt难兼顾所有场景
串行瓶颈  → 多步骤任务耗时长，独立子任务只能排队执行
```

### 9.2 七种协作模式

**模式1：主从委托（Orchestrator-Worker）**
主agent拆解任务，分配给多个独立worker并行执行，汇总结果。适合可明确拆解为独立子任务的场景。并行执行快、上下文隔离，但orchestrator是单点瓶颈。

代表产品：Claude Code（subagent）、本项目（DelegateManager）、AutoGen

**模式2：接力传递（Pipeline）**
每个agent的输出是下一个的输入，如需求→设计→编码→测试→部署。流程清晰可控，但串行执行慢，一步出错后面全受影响。

代表产品：GitHub Copilot Workspace、传统CI/CD pipeline

**模式3：对等讨论（Debate）**
多个agent从不同视角讨论同一问题，主持人判断是否达成共识。多角度思考质量高，但Token消耗极高，收敛时间不确定。

代表产品：ChatDev、CAMEL、MetaGPT

**模式4：层级分治（Hierarchical）**
CEO管VP，VP管总监，总监管一线。适合超大型工程，但层级增加延迟，容易"传话走样"。

代表产品：MetaGPT（CEO/CTO/PM/Engineer角色）、企业级Agent编排平台

**模式5：竞争选优（Competition）**
同一问题分发给多个agent分别解答，Judge评估选最优。结果质量高、鲁棒性强，但Token成本是单Agent的N倍。

代表产品：AlphaCode（生成大量代码候选→过滤选优）、self-consistency

**模式6：评估反馈（Generator-Critic）**
生成者产出，评估者给反馈，生成者改进，循环直到达标。输出质量持续提升，适合创作和代码生成任务。

代表产品：Reflexion、Self-Refine、Constitutional AI

**模式7：动态路由（Router）**
根据用户意图路由到对应专家agent，每个专家有专属system prompt和工具集。响应快，容易水平扩展，但跨领域问题需要在agent间转交。

代表产品：各大厂的智能客服、本项目的Skill advertise→load本质上也是一种路由

### 9.3 七种模式综合对比

| 模式 | 并行性 | 通信开销 | 实现复杂度 | Token成本 | 最佳场景 |
|------|--------|----------|------------|-----------|----------|
| 主从委托 | 高 | 低 | 中 | 中 | 可拆解的独立子任务 |
| 接力传递 | 无 | 中 | 低 | 低 | 有顺序依赖的流水线 |
| 对等讨论 | 低 | 高 | 高 | 极高 | 决策/方案选择 |
| 层级分治 | 高 | 高 | 极高 | 高 | 超大型工程项目 |
| 竞争选优 | 高 | 低 | 中 | 高（N倍） | 高可靠性要求 |
| 评估反馈 | 无 | 中 | 中 | 中高 | 创作/生成类任务 |
| 动态路由 | 无 | 极低 | 低 | 低 | 意图明确可分类 |

### 9.4 本产品的组合方案

```
层次1：动态路由 → Skill advertise阶段，用户请求匹配skill，路由到对应能力
层次2：主从委托 → DelegateManager，复杂任务拆解并行子agent执行后汇总
层次3：评估反馈（隐式）→ ReAct循环中LLM自评估工具结果是否满意
```

### 9.5 SubAgent的完整设计

**子agent创建**：继承主agent的LLM，有独立的system prompt描述子任务目标和上下文，独立的迭代预算（默认50轮）和TodoStore。

**工具权限隔离**：

| 工具 | 是否允许 | 原因 |
|------|----------|------|
| delegate_task | ✗（leaf角色） | 防止递归委托 |
| memory | ✗ | 防止并行子agent产生冲突写入全局持久状态 |
| clarify | ✗ | 子agent在线程池中运行无法直接与用户交互，允许则会永久阻塞 |
| load_skill | ✓ | 通过全局registry继承 |
| MCP工具 | ✓ | 通过enabled_toolsets继承 |
| todo | ✓ | 独立TodoStore实例 |

**四维资源控制**：

| 控制维度 | 默认值 | 目的 | 实现方式 |
|----------|--------|------|----------|
| 并发控制 | 最多3个子agent | 防止API并发耗尽 | ThreadPoolExecutor(max_workers=3) |
| 迭代预算 | 独立50轮 | 不消耗主agent配额 | 独立IterationBudget实例 |
| 超时保护 | 600秒 | 防止子任务挂死 | future.result(timeout=600) |
| 深度限制 | 最多1层委托 | 防止无限嵌套 | current_depth >= max_depth时强制leaf |

**执行结果结构化**：包含goal、success、final_answer、tool_calls_count、iterations_used、tokens_used、duration_seconds。主agent可据此判断子任务质量——如一个子agent用了50次迭代（预算耗尽）但声称成功，主agent应该怀疑结果完整性。

## 十、Harness（稳定运行保障框架）

### 10.1 产品价值

如果Agent Loop是"大脑"，Harness就是"免疫系统+骨骼+皮肤"。一个没有Harness的Agent就像没有保护装置的赛车手：晴天路况好时跑得飞快，但遇到任何意外就是灾难。

LLM的工具调用格式错误率约5-10%，幻觉率更高，没有Harness的Agent跑不过24小时。Harness是Agent从"能跑"到"能用"的关键分水岭。

业界代表性开源Harness项目：

| 项目 | 组织 | 定位 |
|------|------|------|
| DeerFlow | ByteDance | "Super agent harness"——编排子Agent+记忆+沙箱 |
| DeepAgents | LangChain | "Batteries-included agent harness"——文件系统/子Agent/上下文/记忆 |
| SWE-agent | Princeton NLP | 代码修复Agent + YAML驱动的可配置harness |
| Parlant | Emcie | "Interaction control harness"——上下文工程+行为治理 |
| OpenHarness | HKUDS(港大) | "One command to launch all agent harnesses"——CLI agent通用运行时 |
| saifctl | Safe AI Factory | "Safety harness"——收敛循环+Gate/Reviewer/Holdout三验证 |

趋势：Harness正在成为Agent架构中与Agent Loop平级的一等公民。

### 10.2 六大子系统

**子系统1：错误分类与恢复**

核心原则：分类先于处理。不同错误类型对应完全不同的恢复策略。

详见Agent Loop章节错误分类与重试部分。

**子系统2：上下文工程**

四阶段渐进式干预：

| 阶段 | 触发条件 | 操作 | 信息损失 |
|------|----------|------|----------|
| 阶段1：预警 | 60%窗口 | 标记冗余区域，准备压缩 | 无 |
| 阶段2：压缩 | 70%窗口 | LLM生成结构化摘要替换早期消息 | 低 |
| 阶段3：截断 | 80%窗口 | 工具返回只保留摘要，删除原始内容 | 中 |
| 阶段4：紧急 | 90%窗口 | 仅保留system prompt + 最近3轮 | 高 |

Parlant的创新思路：主动做上下文窄化（Context Narrowing）——根据当前对话轮次的主题，只注入相关的规则、知识和工具描述，从源头控制token消耗。这是从"事后压缩"到"事前精选"的范式转变。

**子系统3：迭代控制**

三层防护：

```
Layer 1：硬性预算（Hard Budget）
  - 最大迭代次数（如200轮）
  - 最大Token消耗（如$5/次）
  - 最大执行时间（如30分钟）
  → 任一触发 = 强制终止 + 输出当前摘要

Layer 2：模式检测（Pattern Detection）
  - 重复检测：同一工具+同一参数连续调用N次
  - 震荡检测：A→B→A→B反复切换状态
  - 空转检测：连续N轮无新信息/无副作用
  → 触发 = 注入反思提示/强制re-plan/终止

Layer 3：进展评估（Progress Evaluation）
  - 里程碑机制：每N轮评估"离目标更近了吗"
  - 衰减因子：每轮的价值递减，超过阈值说明边际收益不足
  - 置信度：LLM自评"我有多确定能完成" < 阈值则终止
  → 触发 = 交还控制权给用户 + 附带进展报告
```

SWE-agent的做法：设置cost limit（$3/实例），超出后强制停止并提交当前最优patch，保证不因单个困难实例耗光全部预算。

DeerFlow的做法：引入"迭代衰减"机制——前10轮每轮权重1.0，之后每轮权重递减0.95^n，累积衰减权重低于阈值时判定为"边际收益不足"，自动终止。

**子系统4：工具治理**

工具调用完整生命周期：

```
① 参数校验&清洗（JSON Schema/必填参数/类型转换/注入检测）
② 权限检查&审批（白名单黑名单/路径范围/危险操作需审批/并发频率限制）
③ 参数修复容错（大小写归一化/camelCase转snake_case/模糊匹配/缺失参数填默认值）
④ 沙箱执行（Docker容器/WASM沙箱/文件系统虚拟化/seccomp系统调用限制）
⑤ 结果验证&后处理（返回大小限制/敏感信息脱敏/格式标准化）
```

声明式策略引擎（avakill方案）：通过YAML定义工具策略，在执行前拦截判定，安全规则外置为配置而非硬编码。

**子系统5：安全防护**

Agent面临独特的安全威胁——间接提示注入（工具返回的内容本身可能是攻击载荷）：

```
攻击面1：输入侧 → 用户直接注入，对抗性prompt绕过限制
攻击面2：工具返回侧（最危险）→ 网页/文件/API响应中嵌入"Ignore previous instructions..."
攻击面3：输出/持久化侧 → 输出泄漏system prompt、凭证信息出现在回复、记忆投毒
```

防护矩阵：

| 威胁 | 检测方法 | 防护位置 |
|------|----------|----------|
| 间接提示注入 | 模式匹配+语义分类器 | 工具返回后、送入LLM前 |
| 凭证泄漏 | 正则（API Key/Token模式） | 输出流过滤 |
| 路径穿越 | 路径规范化+白名单（realpath比较） | 文件操作工具前 |
| 命令注入 | Shell元字符检测 | Bash工具参数清洗 |
| 记忆投毒 | 内容可信度评估 | Memory write前 |

核心原则：Defense in Depth，输入、处理、输出三层各设独立检测，任何一层失守后面的层仍能拦截。

**子系统6：可观测性**

三支柱：

```
Logging（日志）→ 每轮决策记录/工具调用入参出参/错误上下文/安全事件审计
Tracing（追踪）→ 端到端请求链路/父子Agent关系/工具调用耗时/压缩降级事件
Metrics（指标）→ Token/轮/成本/成功率/重试率/延迟分布/循环检测触发率
```

追踪维度层级：`Session → Turn → LLM Call → Tool Call → Sub-Agent`

代表工具：LangSmith / Langfuse / OpenTelemetry / Helicone

### 10.3 Harness七条设计原则

| 原则 | 含义 | 反面教材 |
|------|------|----------|
| 分类先于处理 | 必须先诊断错误类型才能选正确策略 | 所有错误一律重试3次 |
| 渐进式降级 | 从温和到激进，逐级响应 | token超限直接清空历史 |
| 静默优先 | 能内部恢复的异常不暴露给用户 | 每次retry都弹toast |
| 预算有限 | 任何自动恢复都必须有上限 | 无限重试直到成功 |
| 可观测 | 静默≠无记录，所有操作必须可审计 | 静默吞掉异常无日志 |
| 最小权限 | 工具权限按需授予，危险操作需审批 | 所有工具默认全权执行 |
| 声明式策略 | 安全规则外置为配置，非硬编码 | 每个安全检查写死在代码里 |

### 10.4 Harness演进趋势

Harness正在经历从"各家自建"到"标准化框架"的演进，成为Agent技术栈中不可或缺的一层：

```
阶段1（早期）：各家手写重试/错误处理，散落在业务代码里
阶段2（现在）：独立Harness框架出现，功能标准化，可插拔
阶段3（趋势）：Harness与Agent Loop平级，成为一等公民
            → 声明式策略引擎（配置而非代码）
            → 自适应恢复（根据历史错误模式自动调整策略）
            → 标准化可观测协议（OpenTelemetry for Agents）
```

## 十一、工具模块实现

### 11.1 通用性抽象设计

所有能力均通过工具体现，核心分为三个模块：

```
Schema   → 与LLM交互的工具定义（JSON Schema格式）
Registry → 工具的注册与管理（全局工具注册表）
Handler  → 工具的实际执行逻辑
```

Registry负责维护工具元数据：名称、schema、handler、可用性检查函数、是否异步、所属工具集、描述。注册时支持覆盖（用于动态更新工具能力）。

### 11.2 工具执行进度预览

工具执行过程中，ToolDispatcher会生成可读的进度预览，通过callback推送到前端，让用户感知当前在执行什么：

```
delegate_task   → "正在委托子代理执行 — {goal前40字}"
memory(add)     → "正在写入记忆 user"
memory(replace) → "正在更新记忆 memory"
todo            → "正在更新3个任务" / "正在读取任务列表"
web_search      → "正在搜索网页 — {query前30字}"
searchDocChunk  → "正在搜索文档 — {query前30字}"
其他工具        → "正在调用 {tool_name}"
```

这个细节确保了即使Agent需要执行耗时操作，用户也能感知到系统在工作，而不是面对空白等待。

## 十二、Claw：本地化Agent

### 12.1 产品定位

上述讨论的Agent都运行在服务端，用户通过API或Web界面远程交互。2024-2025年出现了一个根本性的转变——Agent以本地客户端的形式运行在用户的电脑上，直接操控用户的工作环境。

技术实现上与服务端Agent差异不大，核心是**Agent跑在本地**，可以直接获取本地文件、操作本地程序、执行本地命令，无需用户手动复制粘贴。

### 12.2 服务端Agent vs 本地Claw Agent

| 场景 | 服务端Agent（传统） | Claw Agent（本地） |
|------|---------------------|-------------------|
| "帮我修复这个bug" | "你需要在第15行改成..." → 用户手动改 | 直接编辑文件→运行测试→确认修复 |
| "帮我部署到测试环境" | "执行以下命令:..." → 用户复制粘贴 | 直接执行docker build→push→deploy |
| "帮我创建一个PR" | "去GitHub创建PR，标题是..." → 用户操作 | git add→commit→push→gh pr create |
| "帮我查个bug原因" | "可能是XX问题" → 用户自己去看日志 | grep日志→读代码→定位根因→给出结论 |

### 12.3 核心价值

Claw Agent消除了"告诉用户怎么做"和"用户实际执行"之间的断层——它直接动手做，形成完整的"决策-执行-验证"闭环。

这也是为什么Claude Code能如此高效：它不是在聊天框里告诉你改哪行代码，而是直接编辑文件、运行测试、提交代码，整个软件开发闭环在本地完成。

## 十三、整体架构串联

### 13.1 一次完整请求的全链路

```
用户发送消息
    ↓
记忆模块：预取相关记忆 → 构建<memory-context>围栏 → 注入messages
    ↓
Skill模块：system prompt中有skill广告（已在启动时注入）
    ↓
Agent Loop启动：

  第1轮：
  ├─ 迭代预算检查
  ├─ 上下文压缩检查（首轮通常不触发）
  ├─ THINK：LLM看到消息+tools（含skill广告+基础工具）
  ├─ LLM决策：匹配到某个skill → 调用load_skill
  └─ ACT：加载skill完整指令 + 注入skill对应的MCP工具

  第2轮：
  ├─ THINK：LLM看到skill操作手册 → 按步骤执行
  ├─ ACT：调用MCP工具执行实际操作
  └─ Harness：工具名修复/参数修复/权限校验/执行/结果验证

  第N轮（复杂任务）：
  ├─ 压缩触发：75%窗口 → 三步压缩 → todo状态注入
  └─ 或委托触发：delegate_task → 子agent并行执行

  最终轮：
  ├─ LLM无tool_calls → DONE
  ├─ 流式输出（StreamingContextScrubber过滤围栏标签）
  ├─ 异步触发记忆提取（后处理，不阻塞用户）
  └─ 返回用户
```

### 13.2 能力演进路径

本产品的设计本质上是一个**渐进式能力沉淀系统**：

```
冷启动期：
  只有基础工具和空白记忆，像个聪明的新员工

使用一段时间后：
  - MEMORY.md积累了环境事实
  - USER.md沉淀了用户画像
  - feedback记忆触发L2规则聚合
  - 规则积累触发L3技能生成
  - 技能目录不断扩充

成熟期：
  - 复杂业务场景有专属skill
  - 每次对话加载对应skill按SOP执行
  - 用户不再需要重复交代任何背景
  - 真正实现"越用越懂你"
```

## 十四、产品迭代路径与核心洞察

### 14.1 迭代优先级

**短期（0-3个月）**
先跑通基础Loop、记忆写入和预取、错误重试和压缩这三个核心机制，是整个系统的生命线，优先级最高。

**中期（3-6个月）**
引入Skill系统，把高频业务场景沉淀成skill。建立可观测性体系（没有observability就是盲飞）。建立badcase自动归因流程。

**长期（6个月+）**
记忆到技能的自动转化流水线，让系统真正实现"越用越懂你"。Multi-agent并行能力，处理更复杂的业务场景。建立完整的安全防护体系。

### 14.2 两个最重要的认知

**认知一：观测比开发更重要**

Agent不像传统Java工程——写完、测完、上线就算交付了。它本质上是概率系统嫁接在确定性工程上的产物：今天跑得好好的链路，明天模型升级一个版本，可能tool calling的格式变了、推理偏好漂移了，一批业务场景就悄悄退化了。

这意味着建立完善的可观测体系（轨迹录制、成功率统计、badcase自动归因），持续挖掘退化case、持续打磨prompt和容错逻辑——这不是上线后的"维护"，而是Agent工程的常态。

**认知二：保持对前沿方案的敏感度**

这个领域半年一个代际，一个好的架构思路（比如渐进式加载、结构化压缩）往往比堆人力调参有效10倍。与其闷头优化旧方案，不如花20%的时间看看社区在做什么。

当前Agent并没有跑出一个很划时代的范式，大家还在探索阶段——这既是挑战，也是先发优势的窗口期。
